Chapter 1

Claude Code Primary Execution Boundaries and GPT Codex Secondary Patch Handoff Protocol

Why this topic matters

It matters at branch creation, issue drafting, bugfix intake, refactor start, ops change start, and any moment a second AI tool is about to be used on an active repository task.

Failure patterns
  • Risk: parallel implementation creates incompatible edits across 6 to 12 files -> Mitigation: freeze both paths, keep one authoritative branch, convert the other into proposal notes, and write a merge decision log before resuming.
  • Risk: a small Codex patch silently becomes an architecture change -> Mitigation: enforce the file-count and context-expansion thresholds and reassign to Claude Code the moment scope spills beyond the bounded patch definition.
  • Risk: undocumented reasoning causes recurring bugfix rework 2 weeks later -> Mitigation: require reproduction notes, handoff records, and a post-fix root-cause summary in the PR or issue record.
  • Risk: remote/mobile use introduces a second implementation stream without visibility into desktop branch state -> Mitigation: limit away-from-desk activity to review, triage, issue editing, and pre-authorized bounded requests only.
  • Risk: Codex-originated changes touch schema, auth, secrets, or deployment behavior without repository-aware verification -> Mitigation: mark the branch control-broken, require Claude Code verification, and block merge until ownership and verification are restored.
  • Risk: sunk-cost bias keeps the builder reconciling dual-AI output for hours -> Mitigation: if 90 minutes pass without a clean authoritative path, reset the branch and restart under explicit ownership.

Chapter 2

Repository Layout, Issue-to-PR Command Flow, and Decision Log Storage for GitHub as System of Record

Why this topic matters

It matters the moment a product concept, bugfix, automation, or refactor moves beyond rough capture into more than 30 minutes of execution or begins affecting multiple files, production behavior, dependencies, or data.

Failure patterns
  • Risk: scattered-context repository where code is in GitHub but specs, tasks, prompts, and rationale are elsewhere -> Mitigation: freeze feature work, create missing /docs/specs, /docs/tasks, and /docs/decisions within 24 hours, and backfill the last 3 to 5 major decisions.
  • Risk: unlinked branch drift from generic branch names such as test or update -> Mitigation: block merge until an issue is created and the branch is retroactively mapped to it.
  • Risk: missing decision records for hosting, auth, migration, or deployment changes -> Mitigation: halt merge until a /docs/decisions entry captures context, decision, consequences, affected paths, related issues/PRs, and review trigger.
  • Risk: PR inflation leading to weak self-review and poor AI review signal -> Mitigation: split by concern into separate feature, refactor, and ops PRs with independent verification steps.
  • Risk: mobile or remote operator cannot determine safe action during a failing deploy -> Mitigation: limit remote response to status acknowledgment and defer implementation until desktop if issue, PR, decision entry, or rollback note is missing.
  • Risk: label taxonomy becomes overhead in a solo repository -> Mitigation: keep roughly 20 or fewer active labels and collapse redundant categories.
  • Risk: board state drifts from issue metadata -> Mitigation: use labels and filters as the authoritative state layer so GitHub mobile and notifications still preserve meaning.
  • Risk: README fails as a resumption surface -> Mitigation: rewrite it to show repository purpose, current status, bootstrap command, main docs paths, deployment surface, and decision log location in under 2 minutes of reading.
